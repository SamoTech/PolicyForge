# AL-004 — AppLocker Publisher Rules (Publisher-First Strategy)

**ID:** AL-004  
**Category:** AppLocker / Publisher Rules / Best Practice  
**Risk Level:** 🟠 Medium  
**OS:** Windows 10 Enterprise/Education, Windows 11  
**Source:** Microsoft AppLocker Documentation · CIS Benchmark

---

## Policy Path

```
Computer Configuration
  └─ Windows Settings
       └─ Security Settings
            └─ Application Control Policies
                 └─ AppLocker
                      └─ Executable Rules
                           └─ Create New Rule > Publisher
```

## PowerShell

```powershell
# Create a Publisher rule from an existing file
$rule = New-AppLockerPolicy -FileInformation (Get-AppLockerFileInformation -Path "C:\Program Files\App\app.exe") `
                             -RuleType Publisher `
                             -User "Everyone" `
                             -RuleNamePrefix "Allow"

# Generate publisher rules for all files in a folder
Get-AppLockerFileInformation -Directory "C:\Program Files\MyApp\" -Recurse |
    New-AppLockerPolicy -RuleType Publisher -User "Everyone" -RuleNamePrefix "MyApp"

# Get file publisher info for a specific executable
Get-AppLockerFileInformation -Path "C:\Windows\System32\notepad.exe" | Format-List

# Export generated rules to XML
Get-AppLockerFileInformation -Directory "C:\Program Files\" -Recurse |
    New-AppLockerPolicy -RuleType Publisher -User "Everyone" |
    Set-AppLockerPolicy -Merge
```

## Rule Type Comparison

| Rule Type | Durability | Maintenance | Security | Use When |
|---|---|---|---|---|
| **Publisher** | ✅ Survives updates | Low | High | Signed apps — always prefer |
| **Path** | ⚠️ File-system dependent | Medium | Low (if user-writable path) | Unsigned apps in fixed paths |
| **Hash** | ❌ Breaks on any update | High | Highest | Legacy/unsigned, no path option |

## Description

Publisher rules are the recommended rule type for AppLocker. They identify executables by digital signature (publisher name, product name, file name, and version) rather than file path or hash. This means publisher rules survive application updates and file moves without requiring policy changes. Publisher rules also provide more meaningful policy documentation — "Allow Microsoft Corporation / Office 365" is far more readable than a file hash.

**Publisher rule granularity (most → least specific):**
```
1. Publisher + Product + File Name + Version    (most specific)
2. Publisher + Product + File Name
3. Publisher + Product
4. Publisher only                               (least specific — avoid for 3rd party)
```

## Impact

- ✅ Survives application updates without rule changes
- ✅ Readable, maintainable policy
- ✅ Works across file moves and reinstalls
- ⚠️ Only works for digitally signed files
- ℹ️ For unsigned apps, fall back to Path rules in non-user-writable locations, then Hash
