# AL-007 — AppLocker DLL Rules (Advanced — Enable Only When Ready)

**ID:** AL-007  
**Category:** AppLocker / DLL Rules / Advanced  
**Risk Level:** 🔴 Critical (HIGH false positive risk if misconfigured)  
**OS:** Windows 10 Enterprise/Education, Windows 11  
**Source:** Microsoft AppLocker Documentation

---

## Policy Path

```
Computer Configuration
  └─ Windows Settings
       └─ Security Settings
            └─ Application Control Policies
                 └─ AppLocker
                      └─ (DLL Rules — not shown by default)

; To show DLL rule collection:
AppLocker MMC > Action > Properties > Advanced tab
  └─ Enable the DLL rule collection ✓
```

## PowerShell

```powershell
# Generate DLL rules from existing approved DLLs on reference machine
Get-AppLockerFileInformation -Directory "C:\Program Files\" -Recurse -FileType Dll |
    New-AppLockerPolicy -RuleType Publisher -User "Everyone" -RuleNamePrefix "DLL"

# Audit DLL events (very high volume — start here)
Get-WinEvent -LogName "Microsoft-Windows-AppLocker/EXE and DLL" |
    Where-Object { $_.Id -in @(8003, 8004) } |
    Select TimeCreated, Message |
    Export-Csv C:\DLLAuditEvents.csv -NoTypeInformation

# WARNING: This will produce thousands of events
# Filter to user-writable path DLLs only:
Get-WinEvent -LogName "Microsoft-Windows-AppLocker/EXE and DLL" |
    Where-Object { $_.Message -match "AppData|Temp|Downloads" } |
    Select TimeCreated, Message
```

## Description

DLL rules extend AppLocker to control which DLL files can be loaded. This closes the DLL hijacking attack path where malware drops a malicious DLL with the same name as a legitimate one in a user-writable directory. However, DLL rules are the most complex and high-maintenance AppLocker collection — you must create allow rules for **every** DLL that every allowed application loads, or apps will break silently.

**⚠️ CRITICAL WARNING:** DLL rules should only be enabled after:
1. Complete EXE/Script/MSI rules are stable and enforced
2. A dedicated DLL audit period of 30+ days
3. Full dependency mapping for all allowed applications

## Impact

- ✅ Blocks DLL hijacking from user-writable paths
- ✅ Closes a gap that EXE rules alone cannot cover
- ❌ Very high false positive rate if not fully mapped
- ❌ Can break applications that load DLLs from non-standard paths
- ⚠️ Performance impact on high-DLL-load applications (browser, Office)
- ℹ️ Most organizations stop at EXE + Script + MSI rules — DLL rules are optional
