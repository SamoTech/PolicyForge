# FW-003 — Firewall Stealth Mode & Unicast Response

**ID:** FW-003  
**Category:** Windows Firewall / Stealth Mode  
**Risk Level:** 🟠 High  
**OS:** Windows 10+, Windows 11, Server 2016+  
**Source:** CIS Benchmark Windows 11 v3.0 · Microsoft Security Baseline

---

## Policy Path

```
Computer Configuration
  └─ Administrative Templates
       └─ Network
            └─ Network Connections
                 └─ Windows Defender Firewall
                      └─ [Domain/Standard Profile]
                           ├─ Prohibit notifications: Enabled
                           └─ Prohibit unicast response to multicast/broadcast: Enabled
```

## Registry

```
HKLM\SOFTWARE\Policies\Microsoft\WindowsFirewall\DomainProfile
  DisableUnicastResponsesToMulticastBroadcast = 1

HKLM\SOFTWARE\Policies\Microsoft\WindowsFirewall\StandardProfile
  DisableUnicastResponsesToMulticastBroadcast = 1
```

## PowerShell

```powershell
# Disable unicast responses to multicast/broadcast
Set-NetFirewallProfile -Profile Domain,Private,Public `
    -AllowUnicastResponseToMulticast False

# Verify stealth mode (no response to dropped packets)
Get-NetFirewallProfile | Select Name, AllowUnicastResponseToMulticast, NotifyOnListen
```

## Why Stealth Mode Matters

```
Without stealth (unicast responses allowed):
  Attacker sends broadcast probe to subnet
  → Machine responds with unicast reply
  → Attacker discovers active machine via response
  → Machine appears on network map

With stealth (unicast responses blocked):
  Attacker sends broadcast probe
  → Machine drops silently (no ICMP unreachable, no RST)
  → Machine does NOT appear in nmap/masscan sweep
  → Reduces attack surface visibility

Note: Does not hide machine from directed (unicast) probes
Note: Legitimate multicast protocols (mDNS, SSDP) unaffected
      when those specific inbound rules are allowed
```
