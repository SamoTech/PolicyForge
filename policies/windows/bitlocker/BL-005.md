# BL-005 — BitLocker for Removable Drives (BitLocker To Go)

**ID:** BL-005  
**Category:** BitLocker / Removable Drive Encryption  
**Risk Level:** 🔴 Critical  
**OS:** Windows 10+, Windows 11  
**Source:** CIS Benchmark Windows 11 v3.0 · DISA STIG

---

## Policy Path

```
Computer Configuration
  └─ Administrative Templates
       └─ Windows Components
            └─ BitLocker Drive Encryption
                 └─ Removable Data Drives
                      ├─ Deny write access to removable drives not protected by BitLocker: Enabled
                      ├─ Do not allow write access to devices configured in another organization: Enabled
                      └─ Choose how BitLocker-protected removable drives can be recovered: Enabled
```

## Registry

```
HKLM\SOFTWARE\Policies\Microsoft\FVE
  RDVDenyWriteAccess           = 1   ; Deny write to unencrypted USB/removable
  RDVDenyCrossOrg              = 1   ; Deny write to drives from other orgs
  RDVRecovery                  = 1   ; Enable recovery
  RDVEncryptionType            = 2   ; 1=full, 2=used space only (faster for USB)
```

## PowerShell

```powershell
# List all removable drives and their BitLocker status
Get-BitLockerVolume | Where-Object { $_.VolumeType -eq 'Data' } |
    Select MountPoint, VolumeStatus, EncryptionPercentage, KeyProtector

# Enable BitLocker on USB drive (user provides password)
$pwd = Read-Host "USB Password" -AsSecureString
Enable-BitLocker -MountPoint E: `
    -EncryptionMethod AesCbc256 `
    -PasswordProtector $pwd
```

## Removable Drive Attack Scenarios

```
Scenario 1: Data exfiltration via USB
  Employee copies sensitive files to unencrypted USB
  → USB lost/stolen
  → Any computer can read the drive
  → Data breach

With RDVDenyWriteAccess = 1:
  → Windows refuses to write to unencrypted USB
  → User MUST encrypt USB with BitLocker To Go
  → Lost USB: requires password to read

Scenario 2: Malicious USB drop attack
  Attacker drops USB in parking lot
  User plugs in → autorun malware
  (See also FW policies for USB autorun)

Scenario 3: Encrypted USB from another org (RDVDenyCrossOrg)
  Contractor brings USB encrypted with their org key
  → RDVDenyCrossOrg = 1: write blocked (read-only)
  → Prevents data exfil via externally-managed USB
```
