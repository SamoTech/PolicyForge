# URA-010 — Deny Logon as Batch Job / Service for Interactive Accounts

**ID:** URA-010  
**Category:** User Rights Assignment / Service Account Control  
**Risk Level:** 🟠 High  
**OS:** Windows 10+, Windows 11, Server 2016+  
**Source:** CIS Benchmark Windows 11 v3.0

---

## Policy Path

```
Computer Configuration
  └─ Windows Settings
       └─ Security Settings
            └─ Local Policies
                 └─ User Rights Assignment
                      ├─ Deny log on as a batch job: Guests
                      └─ Deny log on as a service: Guests
```

## secedit Template

```ini
[Privilege Rights]
SeDenyBatchLogonRight   = *S-1-5-32-546  ; Guests
SeDenyServiceLogonRight = *S-1-5-32-546  ; Guests
```

## Description

Prevents Guest accounts from being used as scheduled task or service identities. Guest accounts have minimal auditing and accountability — if a service or scheduled task runs as Guest, its actions appear as unauthenticated/anonymous in logs, making incident response and forensics much harder. More critically, a Guest account running a service could be leveraged to execute code with reduced visibility.

## Service Account Best Practices

| Account Type | Use Case | Security Level |
|---|---|---|
| **gMSA** (Group Managed Service Account) | Services on domain-joined systems | ✅ Best — auto-rotating 128-char password |
| **MSA** (Managed Service Account) | Single-server services | ✅ Good — auto-rotating |
| **Dedicated service account** | Legacy/complex services | 🟡 Acceptable with 128-char pw |
| **Built-in service accounts** (SYSTEM, Network Service) | Simple services | 🟡 Acceptable with least privilege |
| **Administrator** | Any service | 🔴 Never |
| **Guest** | Any service | 🔴 Never |
| **Interactive user account** | Any service | 🔴 Never |
